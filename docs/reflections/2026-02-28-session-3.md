## Key Findings

### CLAUDECODE Environment Variable
- Claude Code sets `CLAUDECODE` env var in child processes
- The `claude` CLI binary checks for this and refuses nested invocation
- HMLR's `_call_claude_cli_api_async` correctly strips `CLAUDECODE` from the subprocess env (line 630)
- Testing from within Claude Code requires `env -u CLAUDECODE` to simulate the MCP server's clean environment

### MCP Stdio + Subprocess Interaction
- Piped stdin (printf | server) causes EOF after all messages are written
- If the tool call spawns a subprocess that takes time (claude CLI ~10-30s), the MCP server may not flush the response before the pipe context collapses
- FIFO-based testing (or Claude Code's persistent stdin) avoids this
- This is NOT a bug — it's how stdio MCP servers are designed to work (long-lived connection)

### Embedding Model Cache
- `BAAI/bge-small-en-v1.5` must be downloaded once before the server works
- The code uses `local_files_only=True` so it won't auto-download at runtime
- First-run setup: `uv run python -c "from sentence_transformers import SentenceTransformer; SentenceTransformer('BAAI/bge-small-en-v1.5')"`

## Process Observations

### Testing MCP Servers from Claude Code
The nested session problem means you can't just pipe to the MCP server from within Claude Code without stripping `CLAUDECODE`. A dedicated test script or FIFO approach is needed. Future sessions should use:
```bash
env -u CLAUDECODE uv run python -c "import asyncio; from hmlr.mcp_server import _dispatch_tool; ..."
```
...to test tool dispatch directly, bypassing the stdio transport entirely.

### "Make It Work" vs "Understand Why It Doesn't"
Switching providers was the "make it work" instinct. David's pushback was the right call — understanding the failure mode (stdin EOF + subprocess timing) was more valuable than a workaround.
